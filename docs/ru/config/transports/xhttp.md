# XHTTP: Beyond REALITY

::: danger
XHTTP объединяет в себе несколько новых концепций в области обхода блокировок и иной подход к организации трафика. По этой причине:

_**Настоятельно рекомендуется ознакомиться с [подробным описанием XHTTP](https://github.com/XTLS/Xray-core/discussions/4113) оригинального разработчика**_

- Приступать к детальной настройке xhttp следует, как только Вам удастся ответить на вопрос "чем различаются `packet-up`, `stream-one` и `stream-up`?"
  :::
  ::: tip
  XHTTP настроен по умолчанию, и для _базовой работы_ достаточно указать только поле `path`:

```json
"xhttpSettings": {
  "path": "/yourpath" // Достаточно
}
```

:::
::: tip
XHTTP имеет собственные настройки мультиплексирования, что делает XHTTP **несовместимым** с [Mux.Cool](../../development/protocols/muxcool.md)
:::

XHTTP - это новый протокол транспортного уровня, который даёт возможность создания устойчивых к DPI анализу виртуальных сетей. XHTTP включает в себя инструменты проксирования через CDN, разделение входящего и исходящего трафика по разным соединениям, улучшенную технологию мультиплексирования и конвертацию клиентского трафика HTTP/3 в HTTP/2 и HTTP/1.1 сервера посредством CDN.

## XHTTPObject

`XHTTPObject` соответствует элементу `xhttpSettings` в [`StreamSettingsObject`](../transport.md#streamsettingsobject).

```json
"xhttpSettings": {
    "host": "example.com",
    "path": "/yourpath", // должно совпадать c аналогичным полем в "downloadSettings"
    "mode": "auto",
    "extra": {
        "headers": {
            // "key": "value"
        },
        "xPaddingBytes": "100-1000",
        "noGRPCHeader": false, // stream-up/one, только для клиента
        "noSSEHeader": false, // только для сервера
        "scMaxEachPostBytes": 1000000, // только для packet-up
        "scMinPostsIntervalMs": 30, // packet-up, только для клиента
        "scMaxBufferedPosts": 30, // packet-up, только для сервера
        "scStreamUpServerSecs": "20-80", // stream-up, только для сервера
        "xmux": { // h2/h3 в основном, только для клиента; обратите внимание на раздел, посвящённый xmux
            "maxConcurrency": "16-32",
            "maxConnections": 0,
            "cMaxReuseTimes": 0,
            "hMaxRequestTimes": "600-900",
            "hMaxReusableSecs": "1800-3000",
            "hKeepAlivePeriod": 0
        },
        "downloadSettings": { // только для клиента, указывает путь входящего соединения
            "address": "", // другой домен/IP
            "port": 443,
            "network": "xhttp", // обязательно "xhttp"
            "security": "tls", // можно reality
            "tlsSettings": {
                ...
            },
            "xhttpSettings": {
                "path": "/yourpath", // должно совпадать с аналогичным полем в начале
                ...
            },
            "sockopt": {} // будет заменено на "sockopt" из upstream, если в последнем "penetrate" равно true
        }
    }
}
```

Минимальная конфигурация **с разделением** upstream и downstream потоков выглядит следющим образом:

<!-- Не уверен, что port не имеет значения по умолчанию ---->

```json
"xhttpSettings": {
    "path": "/yourpath", //Должно совпадать с нижним path
    "extra": {
        "downloadSettings": {
            "address": "cdn.downstream.example", // адрес downstream прокси, не обязательно CDN
            "port": 443,
            "xhttpSettings": {
                "path": "/yourpath", //должно совпадать с верхним path
            }
        }
    }
}
```

> `host`: string

<!-- Не уверен, что правильно понял: добавление заголовка host инвалидит конфигурацию? Если да, обязательно об этом написать warninig здесь или, что лучше, в разделе headers --->

Хост, отправляемый в HTTP-запросе, значение по умолчанию — `""`. Если значение на стороне сервера пустое, значение хоста, отправленное клиентом, не проверяется.

Приоритет выбора хоста для отправки клиентом: `host` > `serverName` > `address`.

> `path`: string

<div id="first-path"></div>

Путь, используемый XHTTP в HTTP-протоколе, значение по умолчанию — `"/"`. Обязательный параметр.

Если значение задано, как `/yourpath`, в конец добавляется `"/"`: `/yourpath/`

<!--- Возможно, это только в stream-one? --->

::: warning
Значение `path` должно совпадать с [аналогичным полем в DownloadSettingsObject](#second-path)
:::

> `mode`: string

Режим работы XHTTP. Значение по умолчанию: `"auto"`

Возможные значения: `"auto"`, `"packet-up"`, `"stream-one"` и `"stream-up"`.

При значении `"auto"` сервер принимает любой режим работы клиента. При указании режима — применяется только он.

При значении `"auto"` клиент выбирает режим по следующим критериям:

- Если используется [безопасность транспорта TLS](../transports/tls.md), используется `"stream-up"`
- Если используется [безопасность транспорта Reality](../transports/reality.md) **без** `downloadSettings` в конфигурации xhttp, используется `"stream-one"`
- Если используется [безопасность транспорта Reality](../transports/reality.md) **вместе с** `downloadSettings` в конфигурации xhttp, используется `"stream-up"`
- Во всех других случаях используется `"packet-up"`

::: tip
Различные CDN могут по-разному относиться к разным режимам работы XHTTP, **сильно влияя на скорость** передачи данных

Это связано с разной структурой HTTP запросов, которые используются XHTTP и которые могут кэшироваться согласно локальным политикам CDN

- Отмечается, что `stream-up` имеет **лучшую совместимость** с агрессивными кэширующими политиками некоторых CDN, чем `stream-one`
  <!---- А что там с packet-up? ---->
  :::

> `extra`: any

<!-- Не думаю, что стоит объявлять целую структуру ради этого. И так конфигурация гораздо запутаннее, чем у других транспортов --->

Особая структура, маркирующая ту часть конфигурации, которая может распространяться отдельно от `host`, `path` и `mode`, поскольку содержащиеся в ней поля меняются сравнительно редко.

Необходима для создания упрощённых клиентских конфигураций для конечных пользователей.

> `xPaddingBytes`: int | string

Значение по умолчанию `"100-1000"`

Устанавливает длину случайных заголовков запроса клиента `Referer: ...?x_padding=XXX...` и ответа сервера `X-Padding: X...` для предотвращения проблемы фиксированной длины заголовков.

Сервер проверяет, находится ли длина заголовка в допустимом диапазоне.

> `headers`: map {string: string}

<!-- Только для клиента? --->

Пользовательские HTTP-заголовки, пары ключ-значение, где каждый ключ представляет имя HTTP-заголовка, а соответствующее значение является строкой.

> `noGRPCHeader`: true | false

Только для клиента. Значение по умолчанию `false`.

Только при режимах `stream-up` и `stream-one`

Отключает gRPC заголовок, который используется gRPC в исходящих запросах

::: warning
XHTTP при проксировании через CDN полагается на избегание кэширующих политик.
gRPC заголовок является одним из ключевых способов "уговорить" CDN направить запрос сразу в Origin.

Отключение gRPC заголовка может повлечь радикальное падение качества соединения
:::

> `noSSEHeader`: true | false

Только для сервера. Значение по умолчанию `false`

Отключает SSE заголовок в ответах сервера.

::: danger
В некоторых случаях при использовании режима `stream-one` возникает ситуация, когда сервер отвечает SSE заголовком на gRPC запрос, что является нестандартным поведением и потенциально может нести демаскирующий эффект.
:::

> `scMaxEachPostBytes`: int | string

Значение по умолчанию 1000000

Позволяет также range значения, например, `"500000-1000000"`

Только для режима `packet-up`

Устанавливает максимальный размер POST запроса в байтах.

::: tip
Разные CDN устанавливают свои ограничения на максимальный размер HTTP запросов.

Для значения по умолчанию выбрано ограничение Cloudflare (1 МБ).
:::

> `scMinPostsIntervalMs`: int | string

Только для клиента. Значение по умолчанию `30` (мс)

Позволяет также range значения, например, `"10-50"`

Только для режима `packet-up`

Устанавливает минимальный интервал между инициированием POST-запросов клиентом для одного прокси-соединения.

> `scMaxBufferedPosts`: int

Только для сервера. Значение по умолчанию `30`

Только для режима `packet-up`

Максимальное количество POST-запросов, кэшируемых сервером для одного прокси-соединения. При превышении соединение разрывается.

> `scStreamUpServerSecs`: int | string

Только для сервера. Значение по умолчанию `"20-80"`

Только для режима `stream-up`

Определяет время, через которые сервер отправляет случайную последовательность байтов длиной `xPaddingBytes` для поддержания активности соединения
::: tip
Разные CDN устанавливают свои ограничения на максимальный простой соединения без передачи фактических данных.

Для выбора значения по умолчанию отталкивались от ограничений Cloudflare (100 с).
:::

> `xmux`: [XMUXObject](#xmuxobject)

Только для клиента.

Настройки специфического для XHTTP мультиплексирования.

> `downloadSettings`: [DownloadSettingsObject](#downloadsettingsobject)

Только для клиента.

Настройки downstream. См. ниже.

### DownloadSettingsObject

Этот объект используется для разделения upstream и downstream трафика - одной из наиболее мощных возможностей протокола XHTTP.

Он представляет собой расширение стандартной [Конфигурации транспорта](../transport.md) с некоторыми изменениями. Ниже приведены все отличия от стандартной конфигурации транспорта.

Если что-либо не приведено, как, например, [finalmask](finalmask.md), это значит, что оно работает так же, как и в конфигурации транспорта.

> `address`: string

Адрес сервера, поддерживаются IPv4, IPv6 и доменные имена. Обязательный параметр.

::: tip
Применительно к XHTTP, здесь подразумевается наличие любого отличного от исходного `host` адреса, который может выступать в роли прокси для `host`, прежде всего, альтернативный CDN. Через этот адрес будет идти входящий трафик.
:::

> `port`: int | string

<!-- Не уверен, что port не имеет значения по умолчанию ---->

Порт сервера, обычно тот же, что и порт, прослушиваемый сервером.

> `network`: string

Способ передачи, используемый потоком данных. Обязательный параметр.

Единственное возможное значение: `"xhttp"`

> `xhttpSettings`: [XHTTPObject](#xhttpobject)

Клиентская XHTTP конфигурация, _аналогичная_, но **не обязательно** _идентичная_ той, которая здесь же и рассматривается. Круг замкнулся.

::: tip
В конфигурации XHTTP **внутри** [DownloadSettingsObject](#downloadsettingsobject) своё поле `downloadSettings` уже не требуется.

Рекурсии можно не бояться!
:::

Единственный обязательный параметр приведён ниже.

> `path`: string

<div id="second-path"></div>

Путь, используемый XHTTP в HTTP-протоколе, значение по умолчанию — `"/"`. Обязательный параметр.

Если значение задано, как `/yourpath`, в конец добавляется `"/"`: `/yourpath/`

<!--- Возможно, это только в stream-one? --->

Должно совпадать с [исходным `path`](#first-path)

> `sockopt`: [SockoptObject](sockopt.md#sockoptobject)

Конфигурация [Sockopt](sockopt.md). Такая же, как и в [конфигурации транспорта StreamSettings](../transport.md#streamsettingsobject).

<!-- Тут, возможно, требуется более подробное описание ---->

Важное отличие состоит в том, что если поле `sockopt` **внутри** [конфигурации транспорта](../transport.md) содержит поле `penetrate` со значением `true`, значение `sockopt` в `downloadSettings` переопределяется.

### XMUXObject

Настройки специально разработанного под XHTTP режима мультиплексирования, заменяющего и **конфликтующего** с классическим [Mux.Cool](../../development/protocols/muxcool.md).

::: warning
Значения по умолчанию в XMUX работают **только** в случае, если все поля xmux пустые или имеют значения 0. В противном случае, все незаполненные поля будут иметь значение 0.

- Таким образом, нужно либо вообще не трогать XMUXObject, либо определить значения _всех полей_ конфигурации или **осознанно** оставить их незаполненными
  :::

> `maxConcurrency`: int | string

Значение по умолчанию `"16-32"`.

**Устанавливается только если все параметры XMUX равны 0**

**Конфликтует с `maxConnections`**. Нужно выбрать только одно.

Максимальное количество одновременных прокси-запросов в одном соединении TCP/QUIC. При достижении этого значения core создаст новое соединение для размещения новых запросов.

> `maxConnections`: int | string

Значение по умолчанию `0`(без ограничений).

**Конфликтует с `maxConcurrency`**. Нужно выбрать только одно.

Максимальное количество одновременных соединений. До достижения этого значения каждый новый прокси-запрос будет открывать новое соединение, после чего core начнет переиспользовать существующие.

> `cMaxReuseTimes`: int | string

Значение по умолчанию `0`(без ограничений). Поддерживает ranges, например, `"1000-2000"`

Сколько раз одно соединение может быть переиспользовано. После указанного количества переиспользований новые прокси-запросы на него назначаться не будут. Соединение закроется после завершения последнего внутреннего запроса.

> `hMaxRequestTimes`: int | string

Значение по умолчанию `"600-900"`.

**Устанавливается только если все параметры XMUX равны 0**.

Иначе значение 0(без ограничений)

Ограничивает максимальное количество HTTP запросов на одно соединение TCP/QUIC. В режиме `packet-up` при циклической отправке POST, если это значение превышено, происходит автоматическое переключение на другое соединение TCP/QUIC.

::: tip
Разные веб-сервера устанавливают свои ограничения на максимальное количество запросов в рамках одной сессии.

Для выбора значения по умолчанию отталкивались от ограничений по умолчанию Nginx (1000 запросов на соединения).
:::
::: warning
Счетчик, использующийся при определении числа запросов, основан на HTTP.

Обычно `stream-one` генерирует один HTTP-запрос, `stream-up` — два, а `packet-up` — несколько. Подсчет не строгий, а GET-запросы в Golang имеют автоматический повтор, поэтому не рекомендуется ставить максимальные значения.

_Конкретные_ значения нужно тестировать под _конкретные_ CDN
:::

> `hMaxReusableSecs`: int | string

Значение по умолчанию `"1800-3000"`. **Устанавливается только если все параметры XMUX равны 0**. Иначе значение 0(без ограничений)

Устанавливает максимальное количество секунд, в течение которых может быть использовано соединение TCP/QUIC. По истечении этого времени TCP/QUIC соединение перестанет принимать новые HTTP-запросы и закроется после завершения последнего запроса. В режиме `packet-up` при превышении времени также происходит переключение на другое соединение.

::: tip
Разные веб-сервера устанавливают свои ограничения на максимальную продолжительность одной сессии.

Для выбора значения по умолчанию отталкивались от ограничений по умолчанию Nginx (1 час).
:::

> `hKeepAlivePeriod`: int

Значение по умолчанию 0, что значит интервал в 45 секунд для Chrome H2 или 10 секунд для quic-go H3. Значение -1 отключает keepalive пакеты

Интервал отправки пакетов keepalive клиентом, когда соединение H2 / H3 простаивает.

::: danger
Случайное время отправки keepalive пакетов является нестандартным, а значит, демаскирующим поведением.

Рекомендуется оставить значение по умолчанию.
:::
